Appeal Procedures


Important Notice: All appeals must follow these procedures exactly. Incomplete submissions or non-compliant requests will not be considered.
Step 1: Verify Your Grade
  • Cross-verify your scores:
    • Check detailed test results on the ZINC System
  • Important notes:
    • Only your most recent submission (before deadline) on ZINC was graded
Step 2: Test Your Code
  1. Download the complete set of test files from the Resources section
  2. Place your COMP2211_task.py, mlp_model.keras, cnn_model.keras in the same folder as the test files
  3. Run the relevant test scripts for the cases you want to appeal
Step 3: Prepare the Appeal Report

Create a PDF containing:

  • Test Case Identification:
    • Specific test case numbers (e.g., #4, #16)
    • Corresponding function names and original output on ZINC
  • Technical Justification:
    • Detailed analysis justifying why your code should pass
    • Direct comparison with reference solution (COMP2211_PA2_sol.py)
    • Note: Statements about playground performance alone are insufficient. E.g.
      • "My code got the expected output on the Playground/Expected Output" and
      • "My code passed the given test cases but not hidden test cases" are not acceptable reasons on their own. We use multiple datasets to grade your PA.
Step 4: Submit Your Appeal
  • Upload to: PA2 Appeal Portal
  • Deadline: 8 Dec 2025 Mon 23:59 HKT
  • Late submissions will not be accepted
Important Policies
  • No partial credit for individual test cases
  • Code modifications will not be accepted - we only evaluate your original submission
  • You may email the TAs or post on Piazza for enquiries regarding the appeal instructions. However, TAs will NOT discuss your appeal (nor comment on appeal progress) via email nor Piazza.
  • Final scores will be updated approximately 7 days after appeal deadline (depending on the number of requests received)
  • All post-appeal grades are FINAL
End of Appeal Procedures

Grading Scheme


Test Categories

Three Fundamental Checks:
  1. Accuracy test: Correct algorithm implementation, i.e., your code output should match that of the sample solution.
  2. Model test: Proper use of models and data structures as required by the assignment. Also, the accuracy on the hidden dataset.
  3. Validity test: Proper coding practices, i.e., not using unauthorized libraries or packages, no plagiarism, etc.

Test Groups

Tests #1, 2, 3, 4, 5, 9: Core Functionality
  • Focus: Algorithm accuracy
  • Metrics:
    • Accuracy Test: Direct comparison between derived results and expected results.
    • Validity test: Proper library imports, plagiarism
  • Features:
    • Hidden dataset test cases
    • Error isolation between tests
Monkey Patch to Prevent Error Accumulation
We used monkey patch to prevent error accumulation in the test cases:
  • Each test case is independent, i.e., do not reuse results from previous test cases
  • We substitute your function implementation with the intermediate results from sample solution, if that test case involves previous functions
This means that if your code fails a test case, it will not affect the grading of subsequent test cases.
Tests #6, 7, 8, 11, 12: Model Functionality
  • Focus: Model Implementation
  • Metrics:
    • Model inspection: Compare the structure of the submitted model and the generated model.
    • Accuracy test: Evaluate the accuracy on the hidden dataset.
  • Features:
    • Hidden dataset test cases

“No module named 'google.colab'”

  • The test scripts are designed to run in a standard Python environment. Therefore, any code requiring Google Colab or other unnecessary libraries will not work during testing.
  • If your code fails due to this error, please remove or comment out the relevant code before testing.
  • You can submit an appeal in this case and receive a 20% penalty.
Universal Rules
  • No unauthorized libraries. Standard Python/NumPy syntax required.
  • Only the last valid submission is evaluated.
End of Grading Scheme
